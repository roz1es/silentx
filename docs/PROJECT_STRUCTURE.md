# Структура проекта BrenksChat

Этот файл нужен как карта проекта, чтобы несколько разработчиков быстро понимали, где находится нужная часть мессенджера и кто за какую область отвечает.

## Основные папки

| Папка | Назначение | Ответственность |
| --- | --- | --- |
| `client/` | Веб-версия мессенджера: React, TypeScript, Vite, стили, страницы, компоненты, контексты | Веб-интерфейс |
| `server/` | Backend: Node.js, Express, Socket.IO, авторизация, чаты, сообщения, почта, сессии, хранение данных | API, база данных, безопасность, realtime |
| `desktop/` | Нативное Flutter-приложение для Windows и macOS | ПК-версия |
| `desktop/windows/` | Windows platform runner для Flutter | Windows-сборка |
| `desktop/macos/` | macOS platform runner для Flutter | macOS-сборка |
| `desktop/installer/windows/` | Скрипты и ресурсы Windows-установщика | Windows installer |
| `.github/workflows/` | GitHub Actions: автоматические сборки и CI | Автоматизация |
| `deploy/` | Nginx, systemd, серверные конфиги и вспомогательные файлы деплоя | Деплой |
| `reports/` | Учебные отчёты и документы | Документация |
| `docs/` | Внутренняя документация проекта | Командная работа |

## Мобильное приложение

Пока отдельной папки мобильного приложения в проекте нет. Когда начнётся разработка телефона, рекомендуется создать:

```text
mobile/
├── android/      # Android runner
├── ios/          # iOS runner
├── lib/          # Flutter/Dart код мобильного приложения
├── test/
├── pubspec.yaml
└── README.md
```

Если мобильная версия будет делаться на Flutter, можно переиспользовать часть логики из `desktop/lib`, но лучше не смешивать код сразу. Сначала стоит сделать отдельный `mobile/`, а общие модели/API позже вынести в общий пакет, например:

```text
packages/
└── brenkschat_core/
    ├── lib/
    │   ├── models.dart
    │   ├── api_client.dart
    │   └── socket_service.dart
    └── pubspec.yaml
```

## Рекомендуемое распределение задач

| Разработчик | Основные папки | Примеры задач |
| --- | --- | --- |
| Разработчик ПК-версии | `desktop/`, `desktop/windows/`, `desktop/macos/`, `desktop/installer/` | Windows/macOS интерфейс, установщик, автопроверка обновлений, desktop UI |
| Разработчик мобильной версии | `mobile/` | Android/iOS интерфейс, мобильная навигация, push-уведомления, адаптация звонков |
| Общая зона | `server/`, API-контракты, Socket.IO события | Только через согласование, потому что изменения backend влияют на все клиенты |
| Общий дизайн | `client/src/index.css`, дизайн-система, цвета, иконки | Желательно согласовывать перед изменениями |

## Важные правила структуры

1. Не переносить существующие папки `client/`, `server/`, `desktop/` без отдельной задачи.
2. Не коммитить `node_modules/`, `dist/`, `build/`, `.env`, временные архивы и локальные сборки.
3. Новые платформы добавлять отдельными папками: `mobile/`, `packages/`, `tools/`.
4. Общие изменения API сначала описывать в issue или pull request, потому что они затрагивают веб, desktop и mobile.
5. Если задача меняет сразу `server/` и клиент, в pull request обязательно писать, какие endpoints или socket events изменились.
